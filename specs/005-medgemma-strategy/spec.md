# Feature Specification: MedGemma Inference Strategy (GBNF & Cache Management)

**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Structured Medical Reporting via GBNF (Priority: P1)

As a radiologist, I want the AI to strictly format its responses according to standard clinical sections (Findings, Impression) without deviating into conversational filler, so that I can quickly read and use the report.

**Why this priority**: MedGemma models tend to hallucinate or break character if unconstrained. Enforcing a physical grammar (GBNF) on the `llama-server` reduces hallucinations by 2-3x and guarantees consistent output structures.

**Independent Test**: Can be independently tested by sending an image to the C++ worker and verifying that the output ALWAYS strictly matches the required structure, regardless of the user's conversational input.

**Acceptance Scenarios**:

1. **Given** the C++ worker is processing a query, **When** it generates tokens, **Then** it must strictly adhere to the provided GBNF grammar (starting with "Findings:" and ending with "Impression:").
2. **Given** the user asks a conversational question ("What's up?"), **When** the C++ worker responds, **Then** it still forces the output into the clinical report structure or refuses to answer conversationally.

---

### User Story 2 - KV Cache Isolation and Invalidation (Priority: P1)

As a system administrator, I want the system to efficiently use the KV cache for repeated system prompts but isolate context when different users interact with the bot, so that context bleed does not occur and VRAM is conserved.

**Why this priority**: With `-np 1` (a single slot on the GTX 1060), alternating between different system prompts (e.g., Doctor vs Patient) without clearing the slot causes cache fragmentation and massive performance degradation.

**Independent Test**: Can be tested by simulating two different users with different system prompts querying the bot sequentially and verifying in the server logs that the slot is cleared and re-warmed cleanly.

**Acceptance Scenarios**:

1. **Given** a user with System Prompt A finishes a query, **When** a user with System Prompt B starts a query, **Then** the dispatcher sends a slot reset command (`{"id_slot": 0, "messages": [], "cache_prompt": false}`) before sending the new query.
2. **Given** a user with System Prompt A sends a second query, **When** the dispatcher routes it, **Then** it uses `"cache_prompt": true` and does NOT reset the slot, leveraging the existing KV cache.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The Dispatcher MUST maintain the state of the active system prompt for the `llama-server` worker.
- **FR-002**: The Dispatcher MUST send a slot reset command to the `llama-server` if the incoming request uses a different system prompt than the currently active one.
- **FR-003**: The Dispatcher MUST configure a GBNF grammar definition for standard radiology reports.
- **FR-004**: The Dispatcher MUST inject this GBNF grammar into requests routed to the C++ native worker.

### Constitution Requirements

- **CR-001**: Feature MUST strictly avoid logging sensitive medical data (Principle I).
- **CR-002**: Feature MUST implement graceful error handling for network timeouts (Principle III).
- **CR-003**: Feature MUST securely manage and isolate user context (Principle IV) through explicit KV cache resets.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Zero context-bleed between users with different system prompts.
- **SC-002**: 100% of reports generated by the C++ worker conform to the "Findings/Impression" structure.
- **SC-003**: VRAM usage remains stable without OOM errors during prolonged multi-user sessions due to proper cache flushing.
